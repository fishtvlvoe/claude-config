# 品質檢查規格

## Skill 品質檢查

### 世代 3 完整性檢查（5 項必備）

| # | 元件 | 檢查方式 |
|---|------|---------|
| 1 | `disable-model-invocation: true` | frontmatter 欄位存在 |
| 2 | GATE 對焦 | 有「複述 → 確認 → OK 才往下」結構 |
| 3 | 強制停止點 | 至少 1 個「等用戶確認才繼續」 |
| 4 | 品質檢查機制 | 有對應類型的品質驗證（見下方） |
| 5 | 完成條件 | 有 checkbox 清單 |

### 品質檢查機制（依 Skill 用途選擇）

#### 寫作類（文章、文案、計畫書）

維度評分制 — 自訂 N 個維度，每維度 10 分。

```markdown
### 品質檢查（必須執行）

**N 維度品質評分（總分 N0）：**

| 維度 | 評估標準 | 得分 |
|------|----------|------|
| {{維度1}} | 10 分：{{好的描述}}；1 分：{{差的描述}} | /10 |
| {{維度2}} | 10 分：{{好的描述}}；1 分：{{差的描述}} | /10 |

標準：{{高分}}可發佈 / {{中分}}需修潤 / 低於{{低分}}重寫
```

常用維度參考：
- 直接性（開場是否直切正題）
- 節奏（句子長短變化）
- 信任度（是否尊重讀者智慧）
- 真實性（是否像有經驗的人分享）
- 精煉度（有沒有多餘內容）
- 拉力（段落間有沒有懸念）
- 結構性（邏輯是否清晰）
- 可行性（建議是否可執行）

#### 開發類（寫程式、修 Bug）

測試驅動 — 所有測試通過 + Code Review。

```markdown
### 品質檢查（必須執行）

1. 單元測試全部通過（`composer test` / `npm test`）
2. Code Review 通過（無 CRITICAL 項）
3. 瀏覽器驗證截圖（如適用）
```

#### 部署類（部署、發佈）

多軌驗收 — 並行檢查多個面向。

```markdown
### 品質檢查（多軌並行驗收）

| 軌道 | 檢查項 | 結果 |
|------|--------|------|
| 部署 | 版本號一致、快取清除 | ✅/❌ |
| API | 端點回應正確 | ✅/❌ |
| Log | 無 500/502 錯誤 | ✅/❌ |

任一軌道失敗 → 觸發回滾。
```

#### 分析類（商業分析、競品分析）

結構完整性 — 關鍵區塊全部填寫。

```markdown
### 品質檢查（結構完整性）

- [ ] 所有必填區塊已完成
- [ ] 數據有來源引用
- [ ] 結論有依據支撐
- [ ] 建議可操作（有具體下一步）
```

#### 工具類（路由、格式轉換）

輸出格式驗證 — 輸出符合預期格式。

```markdown
### 品質檢查（輸出驗證）

- [ ] 輸出格式符合預期
- [ ] 無遺漏欄位
- [ ] 邊界情況已處理
```

## Agent 品質檢查

| # | 檢查項 | 說明 |
|---|--------|------|
| 1 | 職責單一 | 一個 Agent 只做一件事 |
| 2 | 不需 GATE | Agent 不應該有對焦步驟 |
| 3 | 有 Checklist | 檢查項用信心閾值分級 |
| 4 | 輸出結構化 | 有明確的輸出格式 |
| 5 | 可被呼叫 | 說明由誰呼叫、怎麼呼叫 |

## Hook 品質檢查

| # | 檢查項 | 說明 |
|---|--------|------|
| 1 | 指令可執行 | Shell 指令語法正確 |
| 2 | 事件正確 | 事件名稱是合法的 Hook 事件 |
| 3 | timeout 合理 | 通常 < 10000ms |
| 4 | 不依賴 AI | 純 Shell，不呼叫 Claude |
| 5 | 失敗安全 | 執行失敗不會阻塞主流程 |
